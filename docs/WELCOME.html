Welcome

svaha

Claude Code that learns from its own track record. One contract + a small scaffold turn a fresh session that starts cold into a loop that picks up exactly where the last one stopped — and gets a little sharper every pass.

New here? This page explains what svaha does, how the pieces fit, and your first ten minutes. If you haven't installed it yet, open INSTALL.html first — or, if you've never used Claude Code at all, start one step earlier with INSTALL-CLAUDE-CODE.html.

The one idea: a single closed loop

Most of the cost of working with an AI isn't the thinking — it's re-introducing yourself every session, and the drift that creeps in across the gaps. svaha closes both with one circuit instead of a pile of features. Each pass leaves the next one better calibrated:

1Decide with convictionA recommendation carries how sure it is and the real tradeoff — not a flat menu.
2Record in the handoffAt wrap-up it freezes a numbered _NEXT — where things stand, decisions + their outcomes.
3Verify at next bootThe next session reconciles the handoff's claims against the actual files before acting.
4A miss becomes a ruleA high-conviction call that didn't pan out gets promoted to a rule that retunes the gate.

Continuity feeds coherence feeds better decisions feeds continuity. That circuit is the product — everything below is a piece of it.

What it does for you

Sessions stop starting cold handoff loop
When context fills up — or you just say "that's it" — the session writes a frozen, numbered _NEXT_NNN.md: where you are, what's next, the decisions made. The next session reads it and resumes. Run several at once; each gets its own numbered slot, so none clobbers another.
It decides ask-vs-act for you Risk Gate
Every action is weighed on three axes at once — conviction (is this right), reversibility (how cheaply it undoes), blast-radius (how much it touches). Cheap, reversible, confident → it just does it. Expensive-to-undo or wide-reaching → it stops and confirms, no matter how sure it is. Money, external writes, deletes, and big diffs always stop.
It blocks the genuinely dangerous guards
A runtime guard denies reads of secret files (.env, keys, .ssh) and catastrophic shell ops (rm -rf ~, curl | sh); a version guard hard-blocks edits to frozen handoffs; a drift guard flags coupled files before a structural edit. You set the permission posture once — reads run silent, the dangerous stuff always confirms.
It gets better calibrated over time optional · operator layer
Decisions are recorded with their conviction; the next boot checks whether they held. A confident call that failed is a calibration miss — and on the second miss of a kind, it's promoted to a rule that makes the next call of that shape more careful. This whole layer is opt-in; delete it for a lean core that still stops on everything dangerous.
It watches its own context budget canary
A transcript-size signal nudges a wrap-up before you hit the wall and lose the thread — a heads-up, not a hard stop, so your explicit "that's it" stays the reliable path.

One word runs it

svaha is driven by a single keyword — you rarely type a slash command. svaha means "so be it / offered into the fire," and where you say it decides what it does:

  • svaha to begin — opens the launchpad and resumes where you left off. (explicit: /session)
  • svaha to a proposal — "yes, go ahead." (a plain yes)
  • svaha to wrap up — freezes the handoff, then svaha seals it with स्वाहा — the receipt that your work is recorded. (explicit: /handoff)

The seal — स्वाहा — is the reply, not your input

You type the plain word svaha. When a wrap actually completes, svaha echoes स्वाहा (Devanagari) back so you know it landed and is recorded. Same word, two directions: you say it to act; it says it to confirm.

Your first ten minutes

The fastest way to feel the loop is to run it once, end to end:

  1. Boot it. In a project, start Claude Code and just say svaha — it opens the launchpad and resumes where you left off. (Or run the explicit command, /session:) 
    /session
    A brand-new project has nothing to resume yet — that's expected. It prints which files it loaded.
  2. Do a small task — fix a bug, add a function. Notice it states decisions with conviction and acts on the safe stuff without nagging, but stops to confirm anything risky.
  3. Wrap up. When you're done (or context is filling), just say svaha again (or "that's it") — it freezes the handoff and seals with स्वाहा, the receipt that it's recorded. (Explicit command, /handoff:) 
    /handoff
    It freezes a numbered _NEXT capturing where you stopped and what you decided.
  4. Start fresh. Open a new session and say svaha again. This time it recovers where you left off — and reconciles its claims against the actual files before doing anything.
  5. That's the loop. Each pass is a little better calibrated than the last. The longer you run it, the more it knows.

Make it yours

Two things to know

1. It's modular. CLAUDE.md is split along the Risk Gate. The core is the safety skeleton + handoff loop. The clearly-fenced Operator layer is the opinionated judgment half (conviction calibration, premortem, weigh-before-you-ask). Keep it for the full self-tuning experience, or delete that one block for a lean core that still stops on everything dangerous.

2. Editing is your call. The permission posture ships frictionless-but-fenced: reads run silently, push/rm/network/secrets always confirm or block, and file edits are granted once at install (or set to ask once per session — your pick, see the snippet header).
