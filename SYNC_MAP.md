# SYNC_MAP — Coupling Map (change X → update Y)

> **The coherence guard.** *If you edit the trigger on the left, the targets on the right go stale or silently break.*
> This is the **maintenance/coupling** view. Its sibling `SYSTEM_MAP.md` is the **data-flow** view — where material lives and how it moves through the system. The two are orthogonal, not redundant — and they are themselves coupled: **edit one's model of the system → reconcile the other.**
>
> **How to use:** before closing a structural edit, scan this map and apply *every* row whose "When you change…" matches what you touched. Rows are independent — a single change often trips several at once (e.g. renaming a path trips the migration, scripts, and docs rows together). Rows marked **†** are **proven-by-incident** — a real drift was caught in the source system this kit derives from; treat them as live tripwires.
>
> **Automated nudge:** a `PreToolUse` drift-guard hook (`bin/drift-guard.py`, wired in `~/.claude/settings.json`) fires whenever a plumbing file is edited and injects the relevant couplings at edit-time. It's a reminder pointing back here, not a substitute for the full map — and it only knows the paths in its own pattern list (see the §3 "add a plumbing file" row).
>
> **This map references only files that ship in this kit.** As your project grows, add rows — each one earns its place after a real drift, not before.

## 2. Ledger & docs chain

| When you change… | Also update… | Why |
|---|---|---|
| Move/rename a `ledger/` file (`CHANGELOG.md`, `session-fixes.md`, `audit-state.md`, `USER_TASKS.md`, `drift-guard-evidence.md`) | `handoff.md`, `reflect.md`, `audit.md`, `_LOADUP §3` map, `CLAUDE.md` Logging + content-placement table, `bin/coherence-check.py` (`CHANGELOG`/`AUDIT_STATE` path constants) | Written/read via hardcoded relative paths by multiple commands and by `coherence-check.py`; moving one orphans every namer. |
| Add a ledger file type, or change what goes where (decisions vs lessons vs fixes vs changelog) | `CLAUDE.md` content-placement table (the authority for what lives where), `_LOADUP §3` map | The content-placement table is the single source of truth for placement; a new ledger file that isn't routed there gets double-stored or lost. |
| Append/merge `CHANGELOG.md` (newest-first, concurrent sessions) | keep the title + intro **above** the first `##` entry; never let an entry land inside the YAML frontmatter | **† Bad concurrent merges once trapped entries inside the frontmatter, compounding on every append.** `coherence-check.py` parses the most-recent `## ` block — a malformed block breaks its pre-logging check. |
| Move/retire a superseded file | move to `_ARCHIVE/` (never `rm`); confirm the replacement landed + the old file is no longer referenced; remove it from `_LOADUP §3` if listed | `CLAUDE.md` no-hard-delete rule — never delete without explicit user instruction. |
| `SYSTEM_MAP.md` — its model of the file set / layers / data flow | `SYNC_MAP.md` (this map), whenever the system's file set / loop changes | They name the same files from two angles; an edit to one's model of the system must be reflected in the other or they diverge. The drift-guard `map-self` / `docs-chain` patterns fire on both. |
| Any **behavior rule** in `CLAUDE.md` that the portable contract also states (continuity, Risk Gate, communication, coding, privacy, model-selection, memory, coherence) | `AGENTS.md` (the cross-tool mirror — re-summarize the changed rule), `README` "Cross-platform" section if the sync protocol itself changes | `CLAUDE.md` is canonical; `AGENTS.md` is a deliberate *summary* of its universal core for tools that don't read `CLAUDE.md`. Edit `CLAUDE.md` first, then re-sync the mirror — never fork the operator-layer wording into `AGENTS.md`. A rule changed in one and not the other is a silent contract fork across tools. |

## 3. Scripts, settings & guardrails

| Edit `bin/drift-guard.py` itself (the `cat()` coupling rows, section labels, `EXCLUDED_PREFIXES`, or `GUARD_KW`) | this map (keep the rows the guard points at coherent with what the hook actually watches) | The guard fires on its own edits (the `guard-self` pattern) precisely so this reconciliation happens. ## 4. Paths & migrations

| When you change… | Also update… | Why |
|---|---|---|
| Change how `<system-dir>` is resolved at runtime, or the `<kit-dir>` / `<system-dir>` split itself (the resolution anchor is `_LOADUP.md`) | the rule is canonical in `CLAUDE.md` "Two paths" + pointed to from `boot.md` / `session.md`; the discovery is *implemented* in four guards — `bin/coherence-check.py` (`discover_system_dir`), `bin/drift-guard.py` (`_discover_system_dir`), `hooks/resume-line-guard.sh`, `hooks/session-end-backstop.sh` (each runs the **same bounded algorithm**: up-scan CWD+ancestors, then a one-level down-scan of CWD's immediate children, anchor `_LOADUP.md`, never recursive); `<kit-dir>` is baked by `setup.sh` + leftover-checked by `bin/doctor.sh` | One contract in five+ places: the prose rule the model follows must match what the four guards actually compute, or the model resolves `<system-dir>` one way while the hooks resolve it another. **All four must keep the identical up-then-down-scan** (the down-scan is what makes the nested-workspace layout resolve) — change one, change all four + the rule. The anchor is `_LOADUP.md` everywhere — change the anchor filename and every discoverer changes together. |
| Move the **kit** clone (`<kit-dir>`) to a new path | re-run `./setup.sh` (it re-bakes `<kit-dir>` into the installed commands, hooks, and `settings.json`), then `bash bin/doctor.sh` to confirm. The kit's *source* embeds no absolute path (it's all the `<kit-dir>` placeholder); the guards resolve their own location from `__file__`, so only the `~/.claude` install copies carry the old path and need re-baking. | `<kit-dir>` is the one baked path. A moved kit leaves the installed `settings.json` / commands pointing at the old absolute location until setup re-bakes them — `doctor.sh` catches it. |
| Move/rename a **project's data** root (`<system-dir>`) | nothing to sweep — `<system-dir>` is resolved at runtime (the dir at/above CWD holding `_LOADUP.md`, else CWD's single immediate child that holds one — see the resolution row above); scripts + hooks self-discover it and the commands carry the placeholder, not an absolute path. Just keep `_LOADUP.md` at the data root. | The point of the split: project data is hardcoded nowhere, so moving a project needs no rename sweep — only the `_LOADUP.md` anchor must stay at the root. |
| Move/rename `bin/` scripts referenced by hooks or commands | `settings.json.snippet`, every command file that invokes them, `_LOADUP §5` operational-tools list | Hooks and commands invoke scripts by hardcoded path; a moved script silently stops running. |

## Housekeeping note

**Rule of thumb:** after any structural edit, `grep -rn` the tree for the old name / path / field / count before closing the session. Most rows above are just specific instances of this one habit.

**SYNC_MAP ↔ SYSTEM_MAP:** if you also maintain a `SYSTEM_MAP.md` (data-flow view), keep the two reconciled — they name the same files from two angles. The drift-guard `map-self` and `docs-chain` patterns fire on edits to either, as a backstop.
